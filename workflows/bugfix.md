# bugfix.md - バグ修正ワークフロー

このワークフローは、プロジェクト内で発見されたバグを修正するための標準手順を定義します。

## 1. 分析 (Analysis)

*   **バグの再現**: 問題を正確に再現する手順を特定し、再現可能なテストケースを作成する。
*   **根本原因分析**: バグの根本原因を特定し、影響範囲を評価する。
*   **既存ドキュメント確認**: 関連する `ARCHITECTURE.md`, `ADR`, `Specs` を確認し、バグが設計意図に反しているか、ドキュメントの記述が不正確であるかを判断する。
*   **AI指示**: 「バグを再現し、根本原因を分析してください。既存の設計ドキュメントとの整合性を確認してください。」

## 2. 計画 (Planning)

*   **ADR作成/更新**: 根本原因がアーキテクチャ上の問題に起因する場合、`docs/adr/` に新しいADRを作成または更新する。
*   **Spec更新**: 既存のSpecが不正確であったり、バグ修正によって変更が必要な場合、`docs/specs/` の関連するSpecを更新する。
*   **タスク分解**: `TASKS.md` に関連するバグ修正タスクを追加し、Current Sprintに配置する。優先度と複雑度を推定する。
*   **AI指示**: 「バグ修正の計画としてADRとSpecを更新し、`TASKS.md`を更新してください。」

## 3. 実装 (Implementation)

*   **ブランチ作成**: `bugfix/<issue-id>-<bug-description>` の命名規則で新しいGitブランチを作成する。
*   **コード修正**: バグの根本原因に対処するコードを修正する。既存のコーディングルール（`.clinerules/02-coding.md`）と型安全性（`mypy`）を厳格に遵守する。
*   **テスト追加**: バグが将来再発しないことを保証するために、新たな単体テストまたは統合テストを追加する。
*   **AI指示**: 「コーディングルールに従ってバグを修正し、再発防止のためのテストを追加してください。」

## 4. 検証 (Validation)

*   **バグの修正確認**: 作成した再現テストケースがパスし、バグが修正されたことを確認する。
*   **回帰テスト**: バグ修正が他の既存機能に悪影響を与えていないことを確認するために、既存のテストスイートを実行する。
*   **品質チェック**: `ruff check`, `mypy src`, `pytest` を実行し、コード品質とテストカバレッジを確認する。
*   **CI/CDパイプライン**: GitHub Actionsでの自動テストがパスすることを確認する。
*   **AI指示**: 「バグが修正されたことを確認し、回帰テスト、コード品質ツール、CI/CDパイプラインで品質基準を満たしていることを検証してください。」

## 5. ドキュメンテーション (Documentation)

*   **CHANGELOG.md更新**: `CHANGELOG.md` にバグ修正のエントリを追加する。
*   **AI指示**: 「`CHANGELOG.md`にバグ修正を記録してください。」

## 6. 完了チェックリスト (Completion Checklist)

- [ ] バグが完全に修正されたことを確認した。
- [ ] バグの根本原因が特定され、対処されたことを確認した。
- [ ] 関連するADRおよびSpecが最新であることを確認した。
- [ ] `TASKS.md` に関連するタスクが完了済みとしてマークされていることを確認した。
- [ ] 新しいテストがバグの再発を防止することを確認した。
- [ ] すべての自動テスト（ユニット、統合、シミュレーション、ストレス）がパスしていることを確認した。
- [ ] コード品質ツール（`ruff`, `mypy`）によるチェックがパスしていることを確認した。
- [ ] バグ修正が既存の `base_policy.md` および `PROJECT_RULES.md` に準拠していることを確認した。
- [ ] レビュープロセスが完了し、必要な承認が得られたことを確認した。
- [ ] `CHANGELOG.md` にバグ修正のエントリが追加されたことを確認した。
