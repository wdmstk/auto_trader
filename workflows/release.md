# release.md - リリースワークフロー

このワークフローは、本番環境への安全なリリースを計画、実行、検証するための標準手順を定義します。

## 1. 分析 (Analysis)

*   **変更内容のレビュー**: リリースに含まれるすべての新機能、バグ修正、リファクタリングの内容と影響を包括的にレビューする。
*   **リスク評価**: リリースに伴う潜在的なリスク（例: バグ混入、パフォーマンス劣化、運用停止）を評価し、軽減策を特定する。
*   **既存ドキュメント確認**: 関連する `ARCHITECTURE.md`, `ADR`, `Specs`, `AGENT.md` を確認し、方針との整合性を検証する。
*   **AI指示**: 「リリースに含まれる変更内容とリスクを分析し、既存のアーキテクチャやポリシーとの整合性を確認してください。」

## 2. 計画 (Planning)

*   **リリース戦略の定義**: DryRun -> Testnet -> Production の段階的デプロイ戦略を具体化する。
*   **ロールバック計画**: 問題発生時のロールバック手順と基準を明確に定義する。
*   **コミュニケーション計画**: 関係者（運用チーム、ビジネスチーム）へのリリース通知計画を作成する。
*   **Go-Liveチェックリスト更新**: `docs/implementation/trading-go-live-checklist.md` を最終リリース要件に合わせて更新する。
*   **AI指示**: 「リリース戦略、ロールバック計画、コミュニケーション計画を策定し、Go-Liveチェックリストを更新してください。」

## 3. 実装 (Implementation) - (該当なし、このフェーズは計画と検証に焦点を当てます)

## 4. 検証 (Validation)

*   **DryRun**: 仮想環境でリリース候補版をデプロイし、主要機能と運用フローが意図通りに動作することを確認する。
*   **Testnet**: テストネット環境でリリース候補版をデプロイし、取引所との実際の連携、注文のライフサイクル、緊急制御が安定して動作することを確認する。
*   **ストレス/シミュレーションテスト**: 必要に応じて、本番に近い負荷やシナリオで追加のテストを実施する。
*   **メトリクス監視**: DryRunおよびTestnetでのメトリクスを収集し、期待されるパフォーマンスと安全基準を満たしていることを確認する。
*   **品質チェック**: `ruff check`, `mypy src`, `pytest` を実行し、コード品質とテストカバレッジを最終確認する。
*   **CI/CDパイプライン**: GitHub Actionsでの自動テストがパスしていることを確認する。
*   **AI指示**: 「DryRunとTestnetでリリース候補版を検証し、ストレステスト、メトリクス監視、コード品質ツール、CI/CDパイプラインで品質基準を満たしていることを確認してください。」

## 5. ドキュメンテーション (Documentation)

*   **CHANGELOG.md更新**: `CHANGELOG.md` にリリースバージョンのエントリを追加する。
*   **運用ランブック更新**: 必要に応じて `docs/implementation/` に運用に関するランブックを更新または作成する。
*   **AI指示**: 「`CHANGELOG.md`と運用ランブックを更新してください。」

## 6. 完了チェックリスト (Completion Checklist)

- [ ] リリースに含まれるすべての変更がレビューされ、承認されたことを確認した。
- [ ] DryRunおよびTestnetでの検証が成功し、すべての品質ゲートを通過したことを確認した。
- [ ] `docs/implementation/trading-go-live-checklist.md` のすべての項目が満たされていることを確認した。
- [ ] ロールバック計画が明確であり、実行可能であることを確認した。
- [ ] 関係者へのコミュニケーション計画が準備されていることを確認した。
- [ ] すべての自動テストがパスし、コード品質ツールによるチェックが問題ないことを確認した。
- [ ] リリースが既存の `base_policy.md` および `PROJECT_RULES.md` に準拠していることを確認した。
- [ ] `CHANGELOG.md` にリリース情報が追加されたことを確認した。
- [ ] 運用監視体制がリリースされたシステムに対応していることを確認した。
